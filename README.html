Frac Campaign Planning Simulator

A Shiny application for planning and evaluating multi-pad hydraulic fracturing campaigns using Monte Carlo simulation, operational risk modelling with consequence propagation, resource-constrained workload analysis, and scenario optimisation.

The simulator is designed for completion engineers, project managers, and operations teams who need to estimate campaign duration, assess uncertainty, evaluate zipper frac strategies, quantify the impact of additional resources, and identify the most cost-effective execution configuration before field operations begin.

The user does not need to edit R code. The application reads CSV input files, runs simulations, generates audit trails, and exports executive planning reports.

Project Overview

Frac campaigns are complex operations involving multiple wells, multiple pads, shared resources, operational uncertainty, and competing schedules.

Traditional planning approaches often rely on deterministic durations and engineering judgment, making it difficult to quantify schedule uncertainty or evaluate alternative execution strategies.

This project provides a data-driven framework for:

  • Campaign duration forecasting (P10 / P50 / P90)
  • Operational risk assessment with consequence propagation
  • Resource planning and bottleneck detection
  • Zipper frac evaluation, including frac tree constraints
  • Campaign acceleration and investment ranking
  • Automated optimum-scenario search
  • Decision support before execution

Main Features

Campaign Planning

  • Conventional and zipper frac simulation, side-by-side comparison with a common seed
  • Multi-pad campaigns, 20 to 40 wells, randomised pad allocation and stage counts
  • Frac tree constraint modelling (swap delays with 2 trees, diminishing benefit at 3 and 4+)
  • Execution modes: Fast (300 runs), Standard (1,000), Audit (2,000 with full traceability)

Resource Modelling

  • Frac fleets, wireline units, CT / cleanout units, milling units, testing units
  • CT spare capacity can optionally support milling (two-pass capacity calculation)
  • Post-frac flowback and testing window per well (7-10 days, configurable)
  • Indicative resource deployment timeline (Gantt-style)

Risk Modelling

  • Full plug-and-perf risk library: plug pressure test failure, screenout, premature plug set, perforation misfire, UPCT failure, cement in casing, cement above plug, isolation plug failure, plus resource and external risks
  • Consequence propagation: each technical risk cascades into induced workload (wireline re-runs, CT interventions, extra milling plugs, testing interventions, extra pumping) rather than a direct delay only
  • All consequence values overridable per-risk from the assumptions CSV
  • User-adjustable risk multiplier

Decision Support

  • Executive KPI dashboard with readiness score and its drivers
  • Critical bottleneck narrative and recommended actions
  • Investment ranking: net benefit and benefit/cost of each proposed resource addition
  • Scenario optimiser: grid search over resource configurations, common-random-number screening, Pareto frontier of duration vs total mobilisation cost, one-click apply of the recommended scenario

Audit and Reporting

  • Executive PDF report (landscape, KPI dashboard, charts, styled tables, deployment timeline)
  • Well-level audit trail, risk event log with consequence columns, assumptions-used table
  • Downloadable audit package (zip) with 18+ CSV exports
  • Input fidelity check: simulated vs historical distributions

Key dependencies as modelled:

  • Wireline gates frac. A stage cannot pump until wireline has perforated and set the plug. In zipper mode, if wireline workload per well exceeds frac workload per well, the frac fleet waits (wireline_readiness_delay_days) and the idle cost is reported.
  • Frac trees gate zipper. Zipper requires 2 trees minimum. With exactly 2, each inter-well transition incurs a swap delay; a 3rd tree reduces transition waiting (~5%), 4+ slightly more (~10%, diminishing).
  • Milling follows frac. Plugs to mill = final stages + replacement plugs (from risks) + consequence plugs (e.g. isolation plug failure).
  • CT can support milling (optional). CT primary duties (cleanout, SCMT, risk interventions) are reserved first; only spare capacity over the frac-path campaign window transfers to milling, at a configurable efficiency (default 0.65).
  • Testing follows milling. Every well needs a flowback + testing window (7-10 days default). Risk events (premature plug set, plug test failure, isolation plug failure) add testing interventions.

Campaign Duration Logic

Per well:
  wireline workload = stages x time/stage + rig up/down + temp logs
                      + contingency + risk delays + induced re-runs
  frac workload     = (stages x time/stage + settling + isolation plugs
                      + risk delays + induced pumping + tree swap delay)
                      x zipper factor + external delays
  CT workload       = cleanout + SCMT + risk delays + induced interventions

Frac path (Pass 1):
  conventional: CT + frac + wireline fleet-days, summed over wells
  zipper:       CT + max(frac, wireline) fleet-days  (wireline overlaps pumping)

CT support for milling (Pass 2, optional):
  spare CT  = frac-path duration x CT units - CT primary workload
  milling   = (gross milling - spare CT x efficiency) / milling units

Campaign duration = max(frac path, milling fleet-days, testing fleet-days)

Scenario Optimiser

The optimiser answers: which configuration delivers the lowest time at the lowest cost?

  • Objective: total mobilisation cost = all contracted units x day rate x P50 duration. This embeds both schedule and idle time, so an extra unit only wins if its schedule saving pays for its contract.
  • Method: every configuration in the grid is screened at reduced iterations with a common random seed (fair like-for-like ranking); the top 5 are refined at full iterations.
  • Output: Pareto frontier of duration vs cost; the recommended scenario is the lowest-cost point on the frontier, with the fastest configuration quoted for contrast. One click applies the recommendation to the sidebar.

Input Files

historical_wells.csv

Historical well performance used to derive operational duration distributions. Required columns include WellID, PadID, StagesCompleted, PlugsInstalled, ContingencyPlugs, FracDays, SCMTDays, MillingDays, FracDaysPerStage, MillingDaysPerPlug.

master_risks_assumptions.csv

Assumptions and risk library. Required: Category, Variable / Risk Event, Type, Probability, Min/Most Likely/Max days, Simulation Impact. Optional consequence-override columns: extra_wireline_runs, extra_ct_days, extra_milling_plugs, extra_testing_days, extra_frac_days.

Templates: data_templates/. Validation reports the exact rows that fail (Min <= ML <= Max ordering, probability in [0,1]).

Outputs

Audit package (zip): simulation_summary, simulation_well_details, simulation_risk_event_log (with consequence columns), resource_utilization, resource_utilization_summary, assumptions_used, executive_summary, executive_kpis, delay_contributors, stage_level_risks, risk_consequences, bottleneck_detection, resource_recommendations, investment_ranking, cost_impact, wireline_constraint, traffic_lights, readiness_score, resource_timeline, management_report.pdf.

Optimiser results export separately from the Optimiser tab.

Limitations

  • Operational planning model: workload and capacity aggregation, not a discrete-event schedule. The deployment timeline is indicative.
  • Not a hydraulic fracture propagation, reservoir, or production model.
  • Risk consequences are deterministic per event (library values); delays are sampled.
  • CT spare capacity is assumed uniformly available across the campaign window.
  • Optimiser recommendations are conditional on the day rates entered; sensitivity-check against contract ranges.
  • Results depend on the quality of historical data and assumptions.
